内存回收优化

写在前面

内存回收 主要包含两个大部分，回收 kernelspace 和 userspace，kernelspace 的内存回收主要包含 slab内存回收 和 percpu的内存回收。userspace的主要是匿名页和文件页的回收。

内存回收异步化

slab 内存回收优化方案一

kswapd 内存回收 与 direct_reclaim 会调用到 shrink_slab，来回收 kernel的 slab
内存，shrink_slab 会调用 各个注册到 shrinker_list 的 shrinker 的回调函数来回收各种各样的内存，比如 gpu md-raid5 等 slab shrinker。

在实际项目中，可能会遇到某个 shrinker写的有问题，持锁时间过长，执行时间过长，导致 shrink_slab 阻塞住，进而进行 direct_reclaim 的线程被阻塞住。

我们将 shrink_slab 放到 shrink_slabd 中去执行，做 direct_reclaim 的线程中只wakeup shrink_slabd。做direct_reclaim 的线程直接被 block住。

风险

如果做direct_reclaim 的线程优先级比较高时 rt的或者 nice 值为负的线程，本来会以比较高的线程优先级去做 shrink_slab的，以后只能在 优先级比较低的 shrink_slabd 中去做了。
这种情况下会降低内存回收的速度。

slab 内存回收优化方案二

一个slab类型的 cache，执行 他的shrinker函数，一般能回收 5-20%的内存。

系统中 shrinker_list 中注册了 shrinker_a shrinker_b shrinker_c三个回调函数，分别对应 slab_a slab_b slab_c 三个 slab。
在不同场景下 各个slab的使用情况也都不一样。
场景a中， slab_a 使用很多有 5000 个 page， slab_b 有 50个 page，slab_c 有 30个page。(视频下载场景， slab_a 是文件系统的 slab cache)
场景b中，slab_a 使用很多有 30 个 page， slab_b 有 50个 page，slab_c 有 1000 个page。(相机场景， slab_c 是相机驱动使用的 slab cache)
假设需要回收64page的内存， 才能满足内存分配的需求。

在场景a 中，如果先执行 shrinker_b shrinker_c，即使回收 50%内存页面，也只能回收 40个page，然后还需要重新执行 shrinker_a，这期间需要分别拿 多次锁，执行多次 do_shrink_slab 过程，如果先执行 shrinker_a，可能不需要执行 shrinker_b, shrinker_c。

同理，在场景b中，如果先执行 shrinker_c，可能不需要执行 shrinker_a, shrinker_b。

我们在 slab_x 分配释放回收时，进行使用的统计，然后对slab的 shrinker进行排序，
这样可以在 后面做 shrink_slab 的时候，尽量快的完成工作，但还是内存回收操作同步进行的。

lru page 页面回收

lru 是指用户空间的匿名页，与kernel space的file cache等，是通过 shrink_lruvec 来回收inactive 页面的。

shrink_lruvec –> shrink_list —> shrink_active_list —-
| |
–> shrink_inactive_list —> isolate_lru_pages

在现有机制中 isolate_lru_pages –> __isolate_lru_page –> trylock_page 失败的页面是不参与本次页面回收的，如果在系统内存非常低的情况下，可能会导致内存回收数量不足，导致分配失败。

现在可以在 trylock_page 失败之后不直接返回，将这些page 加入到另一个 list shrink_lruvecd_list中，
等待active_list or inactive_list 完全扫描结束之后再再起一个异步线程 shrink_lruvecd 来处理刚刚 放到 shrink_lruvecd_list 中的页面，大概率此时通过 trylock_page 可以成功的，因为距离上次尝试已经过去一段时间了。

内存回收负载优化

基于 page->_mapcount 的内存回收负载优化

先看 struct page 中 _mapcount 的描述，_mapcount 指的是 被page table 引用的次数，就是一个物理页面被映射的次数。

lruvec 链表中的页面要老化移动是根据这个 物理页面 最近是否被访问过来确定的。
那么如何确定一个物理页面是否被访问过呢？
需要找出 映射到这个物理页面的所有 struct page 的页面的 vma，看看他vma->vm_flags 来确定这个页面是否最近被访问过，如果被访问过 说明就是 active的，就不需要将其放到 inactive list中。也不需要进行回收。
但是当一个页面 _mapcount 很大的时候，往往刚刚将他放入到 inactive_list 或者后续也将他回收了，后续很快被访问又会放入到 active_list 中，会导致内存颠簸，做一些无用功。

确定一个页面是否被访问过是 page_referenced 来确定的，会进行 rmap_walk，在页面回收过程中rmap_walk 的负载相当的大，而且是浪费在 _mapcount 大于 20的页面上的。
所以直接放弃 _mapcount 大于20的页面的扫描，会大大降低 shrink_active_list 的负载，进而加快内存回收速度。_mapcount > 20 也意味着这个页面比其他页面有着更大的概率下一步会被访问到。

page_referenced -> rmap_walk -> page_referenced_one

其实在 shrink_active_list 这一步 进行了 _mapcount 的保护之后，在 shrink_inactive_list 中 _mapcount > 20 的页面应该很少了。

数据支撑

rmap_walk 过程中 _mapcount > 20 的页面占 10% 左右，rmap_walk 占用耗时占用80%以上。

基于 局部性原理 的内存回收负载优化

在一个 确定一个 struct page 是否最近被访问的时候，

  1. 如果一个页面最近被访问了，那么很可能这个 vma 中的其他相邻的 虚拟地址映射的物理页面也已经被访问了，这些页面就不需要通过 rmap的方式来确定他们 是否 page_referenced了，最好的情况是节省了16个页面的 page_referenced 的调用。遍历 这个 vma中的页面的附近16个页面，看他们是否刚刚刚刚被访问，然后设置 page 的young bit。
  2. 如果一个页面最近没有被访问， 不进行 look_around。

内存分配优化 ux memfirst

linux 内存分配与回收策略是：

  1. mem > high_wmark: 正常分配，不回收
  2. low_wmark < mem < high_wmark: 正常分配， kswapd 启动异步回收
  3. mem < min_wmark: 正常进程开始同步回收

这是原生linux kernel的策略。不管是 ux 还是后台线程 low 谁
