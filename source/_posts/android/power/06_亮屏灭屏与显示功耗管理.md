# 亮屏灭屏与显示功耗管理

在任何移动设备或智能穿戴设备中，**屏幕显示（Display）都是整机功耗占比最大（常年占据 active power 的 40% ~ 70%）的“大户”**。优秀的显示功耗管理，能在维持人眼视觉舒适度的前提下，为系统争取数小时的续航时间。

本节我们将基于 **AOSP 14**，深入分析 `DisplayPowerController`（DPC）的亮灭屏决策流程、自动亮度（Auto-Brightness）过滤算法，以及现代手机的动态刷新率（VRR）与 AOD（息屏显示）功耗调优。

---

## 一、 DisplayPowerController (DPC) 核心逻辑

在 Android 14 中，屏幕亮暗与状态的具体执行并不是由 PMS 直接控制，而是由 `DisplayManagerService` 内部的 **`DisplayPowerController`**（简称 DPC）主导。

### 1. DPC 与 PMS 的交互链路
当 PMS 的 `updatePowerStateLocked()` 运行到阶段 6 时，会向 DPC 投递一个 `DisplayPowerRequest` 请求：

```
 PowerManagerService (Java)
         │  1. mDisplayManagerInternal.requestPowerState(mDisplayPowerRequest)
         ▼
 DisplayPowerController (Java System Server)
         │  2. updatePowerState()
         ├── 2.1. 计算目标屏幕状态 (Display.STATE_ON / STATE_DOZE / STATE_OFF)
         ├── 2.2. 计算目标亮度 (基于 Auto-Brightness, Thermal limits, Dim state)
         ├── 2.3. 处理亮度渐变过渡 (Animate Screen Brightness)
         └── 2.4. 通知 SurfaceFlinger/HAL 更新面板状态
```

### 2. 核心状态转移
DPC 接收请求后，会将屏幕驱动至以下四种核心硬件状态之一（对应 `android.view.Display`）：
* **`Display.STATE_ON`**：屏幕完全点亮，处于正常工作状态。
* **`Display.STATE_DOZE`**：低功耗待机状态（AOD 模式）。屏幕以极低亮度刷新，面板可能采用特定调色板。
* **`Display.STATE_DOZE_SUSPEND`**：比 DOZE 更省电。屏幕只显示静态图像并停止刷新（由面板硬件自刷新），CPU 允许进入休眠，不往屏幕投递帧。
* **`Display.STATE_OFF`**：屏幕完全熄灭，断开面板供电。

## 二、 自动亮度控制与平滑渐变机制

自动亮度（Auto-Brightness）是通过环境光线传感器（Light Sensor）采集的照度（Lux），动态映射出当前最合适的屏幕亮度。其底层处理非常注重人眼的舒适感和功耗的平衡。

### 1. 光线传感器信号过滤与防抖 (Debounce)
光照强度会随用户走动瞬时发生剧烈变化（如穿过树荫、路灯）。为了防止屏幕频繁闪烁（闪烁会带来极差的视觉体验，且频繁升降背光极其耗电），DPC 引入了**防抖动过滤算法（Debounce）**：
* **亮化防抖时间（Brightening Light Debounce）**：当环境突然变亮时，系统不会立刻调亮，而是等待一段时间（如 2000ms），确认不是瞬时闪烁后才开始变亮。
* **暗化防抖时间（Darkening Light Debounce）**：当环境突然变暗时，防抖时间通常设置得更长（如 4000ms），避免用户在稍微偏头挡住光线时屏幕就变暗。

### 2. Spline 亮度插值曲线
在 `AutomaticBrightnessController.java` 中，系统将 Lux 照度转化为屏幕亮度（0.0f 到 1.0f 之间的小数）不是线性的，而是通过一个**三次样条插值曲线（Spline Interpolation）**进行平滑计算：

```java
// base/services/core/java/com/android/server/display/AutomaticBrightnessController.java
// 基于配置的 Lux 数组和 Brightness 数组生成平滑插值曲线
Spline brightnessSpline = Spline.createSpline(luxLevels, brightnessLevels);
float targetBrightness = brightnessSpline.interpolate(filteredAmbientLux);
```

### 3. 亮度平滑渐变（Ramp Animator）
计算出目标亮度后，DPC 不会让屏幕亮度“瞬变”到该值，而是启动一个名为 `RampAnimator` 的动画器，以每秒一定的步长（Step）让亮度“滑行”过去，保证视觉过渡的丝滑感，同时缓解电源管理 IC（PMIC）突然变载的压力。

## 三、 高级显示功耗优化技术 (AOSP 14 演进)

### 1. 动态刷新率（Variable Refresh Rate - VRR）
现代智能手机普遍支持 120Hz、90Hz、60Hz 的刷新率，甚至搭配 LTPO 面板可支持 10Hz 至 1Hz 刷新率。
* **高刷新率（120Hz）**：滚动列表、玩游戏时开启，消耗大量 GPU 渲染和面板功耗。
* **低刷新率（1Hz ~ 10Hz）**：在静态阅读、看电子书、或者 AOD 息屏时开启，直接节约 30% 以上的屏幕电量。
* **AOSP 控制层**：`SurfaceFlinger` 的 `Scheduler` 模块会根据当前图层是否有更新活动，动态修改刷新率，并调用硬件接口将 Hint 传给 Display HAL。

### 2. 显示控温降亮（Thermal Brightness Throttling）
在炎热环境下（或手机正在打游戏发热），屏幕发热与 SOC 发热会相互叠加，导致整机过热。
* `ThermalManagerService` 会持续向 DPC 投递当前的发热等级（Thermal Status）。
* 当发热达到临界值（如 `THERMAL_STATUS_LIMIT_DISCHARGE`）时，DPC 会强行将最大背光限制在如 60% 甚至 40%，以降低背光 LED 的发热量，强制实现热降退。

### 3. AOD 面板自刷新（Self-Refresh - PSR）
当手机进入 AOD 状态并处于 `Display.STATE_DOZE_SUSPEND` 时，
* SoC 的显示引擎（Display Engine）会向屏幕面板的 DDIC（Display Driver IC）芯片发送一帧静态图像（包含时间、通知图标）。
* 随后显示引擎直接关闭，不发送任何刷新信号（VSYNC 信号关闭）。
* 屏幕面板通过 DDIC 内部集成的 GRAM 缓存，自主刷新当前图像（维持 1Hz 甚至更低）。
* 这一技术使 SoC 的显示控制器可以完全休眠，整机待机功耗降低数倍。

## 四、 显示功耗参数配置文件 (`config.xml`)

在定制 Android 系统时，OEM 厂商通常会在 `/device/vendor/` 对应的 overlay 目录中，通过修改 `frameworks/base/core/res/res/values/config.xml` 来优化显示功耗：

* **`config_autoBrightnessLevels`**：定义光照传感器的 Lux 分界点。
* **`config_autoBrightnessLcdBacklightValues`**：定义对应 Lux 分界点下的背光值。
* **`config_screenBrightnessDim`**：系统超时无操作但未灭屏时，屏幕变暗（Dim）状态的亮度（通常设为极小值如 5%）。
