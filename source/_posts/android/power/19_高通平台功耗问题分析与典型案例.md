# 高通平台功耗问题分析与典型案例

在主流 Android 手机厂商中，高通（Qualcomm）骁龙处理器占据了旗舰机型的半壁江山。由于高通平台底层架构复杂，包含了众多的异构微处理器（如 APSS 核心、Modem 蜂窝基带、ADSP 传感器/音频、CDSP 神经网络），其功耗调试不仅局限于 Android 上层，还需要大量使用高通专属的底层分析工具和内核节点。

本节我们将详解高通平台的功耗排查思路、专属工具链，以及一个因系统无法进入 XO Shutdown 的典型功耗案例排查实录。

---

## 一、 高通平台功耗分析“百宝箱”（专属工具与节点）

在高通平台上进行功耗调试，除了通用的 Android 工具外，通常会动用以下底盘武器：

### 1. 核心内核调试节点
* **`/sys/kernel/debug/rpm_master_stats`** 或 **`/sys/kernel/debug/rpmh_stats`**（较新平台）：
  * **用途**：高通平台的低功耗模式（LPM）由底层的 **RPM (Resource Power Manager)** 或 **RPMh** 统一协调。此节点记录了各个 Master 处理器（APSS-应用处理器, MPSS-基带, ADSP-音频DSP, CDSP-计算DSP, TZ-安全世界）各自的深睡计数与睡眠时长。
  * **排查关键**：当系统待机高时，通过该节点检查哪个 Master 处理器一直处于 `Active` 状态，未进入低功耗状态。
* **`/sys/kernel/debug/lpm_stats/stats`**：
  * **用途**：显示 CPU 核心及各个电源域（Power Domain）进入低功耗模式（如 Vdd-min 极低电压状态、Core Collapse 核心坍缩断电）的具体统计。

### 2. 高通专属调试工具链
* **QXDM (Qualcomm Extensible Diagnostics Monitor)**：
  * **用途**：高通射频、Modem 基带最权威的抓包分析工具。用于分析通话、数据传输、eDRX/PSM 状态下的信令交互，是排查网络射频功耗的首选。
* **QPM (Qualcomm Power Monitor)**：
  * **用途**：用于导入物理程控电源（如 Monsoon Power Monitor）测得的电流波形，与 QXDM 抓取的底层信令日志进行时间轴对齐，实现“波形与代码”的完美联动剖析。
* **QDSS (Qualcomm Debug Subsystem)**：
  * **用途**：硬件级的 System Trace 系统，能以微秒级采集芯片内部总线活动及系统时钟状态。

## 二、 高通低功耗睡眠层次 (LPM)

为了实现极低的待机电流（通常基准线为 2mA 左右），高通 Soc 必须在灭屏静置时层层进入以下低功耗睡眠状态：

```
       [ 正常运行 (Active) ] ───> CPU 核心跑在 C0 状态
                  │
                  ▼  所有 CPU 核心闲置 (CPUIdle)
       [ 核心断电 (Core Collapse) ] ───> 各个 CPU 核心分别切断供电
                  │
                  ▼  整机应用处理器空闲，释放所有用户锁
       [ APSS Power Collapse (APSS PC) ] ───> 应用处理器子系统整体断电
                  │
                  ▼  所有 Master 处理器 (MPSS, ADSP, CDSP) 均进入休眠
       [ XO Shutdown (Crystal Oscillator) ] ───> 外部 19.2MHz 主晶振完全关断
                  │
                  ▼  最深睡眠
       [ Vdd-Min ] ───> PMIC (电源芯片) 降压至物理极限，系统维持毫安级电流
```

如果在最后一环中，**主晶振 XO 无法关断（XO Shutdown 计数不增加）**，整机电流通常会被吊死在 **15mA ~ 30mA**，发生严重耗电。

## 三、 典型案例：ADSP 传感器锁死导致 XO 无法 Shutdown 案

### 1. 问题现象
* **测试报告**：某高通骁龙 8 Gen 2 平台手机，在飞行模式下灭屏待机测试中，待机平均电流高达 **18mA**（基线应为 **1.8mA** 左右）。
* **初步排查**：读取 `/sys/power/autosleep` 显示系统已经写入 `mem`，且上层 Android 的 `dumpsys power` 表明没有任何 Java WakeLock 残留，系统挂起（Suspend）流程表面上运行成功。

### 2. 分析步骤与高通节点定位
由于系统名义上已经 Suspend，但物理电流依旧极高，这表明 SOC 底层硬件并没有进入最深的休眠状态（XO 晶振未关断）。

1. **读取高通 Master 状态节点**：
   在手机灭屏待机 10 分钟后，亮屏使用 ADB 抓取 `/sys/kernel/debug/rpm_master_stats` 节点：
   ```bash
   adb shell cat /sys/kernel/debug/rpm_master_stats
   ```
   * **观察输出数据**：
     * `APSS (Application Processor Subsystem)`: `shutdown count: 120, total_time: 580000 ms`（应用处理器成功断电，且时间很长）。
     * `MPSS (Modem)`: `shutdown count: 4, total_time: 550000 ms`（基带也进入了休眠）。
     * **`ADSP (Audio/Sensor DSP)`**: **`shutdown count: 0, total_time: 0 ms`**（ADSP 睡眠计数为 0！表明音频/传感器协同处理器一直处于 Active 激活态）。
     * **`XO (Crystal Oscillator)`**: **`shutdown count: 0`**（外部主晶振完全没有关断过！证实了高功耗的根源）。

2. **锁定 ADSP 底层负载**：
   既然定位到 ADSP 未睡，说明有外设驱动或者传感器子系统（QSensors）一直在高频戳醒 ADSP。
   * 使用高通的 **QDSS** 工具抓取 ADSP 的底层时钟树 Trace。
   * 分析 Trace 发现，ADSP 内部的 `sns_sam_pedometer`（计步器算法服务）时钟一直被强行开启（Clock Locked），并没有释放。

### 3. 根因剖析
* 检查计步器相关的应用与 Framework 交互。
* 发现某款健康类 App 在后台注册了 `Sensor.TYPE_STEP_COUNTER`（计步传感器）。
* 正常情况下，该传感器应交由 Sensor Hub 硬件 FIFO 批处理。但在高通这版定制 BSP 驱动中，重力/加速度计驱动在灭屏后未能正确切入 Low Power 模式，导致传感器芯片（IMU）持续抛出高频物理中断戳醒 ADSP，导致 ADSP 无法深睡，进而把全系统的 XO 主晶振锁死在开启状态。

### 4. 解决方案
* **驱动修复**：更新高通传感器 HAL 及底层的 SSC（Sensor Subsystem）固件，修复灭屏后步数传感器的中断阈值过滤逻辑。
* **功耗优化**：当检测到系统处于休眠状态时，强制将计步采样率下调，并开启 10 秒的 FIFO 硬件缓存，允许 ADSP 深度休眠。
* **效果验证**：修复后，重新读取 `rpm_master_stats`，`ADSP` 的 `shutdown count` 开始快速增加，`XO` 成功实现 `Shutdown`，待机电流骤降至 **1.6mA**，问题彻底解决。
