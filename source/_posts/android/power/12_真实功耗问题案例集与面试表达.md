## case写作模板

```text
场景：
    待机/亮屏/充电/游戏/弱网/定位/蓝牙。

现象：
    电流、掉电、温度、FPS、充电速度等指标异常。

测试条件：
    版本、供电、亮度、网络、温度、起止时间。

证据：
    dumpsys、Perfetto、BatteryStats、wakeup_sources、thermal_zone、电流仪。

根因：
    应用、Framework、HAL、Kernel、vendor或硬件。

修复：
    代码、配置、策略、应用行为或平台参数。

验证：
    同条件前后对比。
```

## Case：USB插线导致待机测试失真

现象：

```text
想分析待机，但设备通过USB连接ADB。
```

证据：

```text
dumpsys battery: USB powered=true
dumpsys power: mIsPowered=true, mPlugType=2, mStayOn=true
```

结论：

```text
当前不是自然待机条件。插USB会改变供电和电源策略，
不能直接用这组数据判断灭屏待机功耗。
```

面试表达：

```text
我会先确认测试条件是否成立，例如是否插USB、是否stay awake、
是否满电或充电。功耗数据如果条件不干净，后续分析会偏。
```

## Case：Thermal HAL不可用

现象：

```text
dumpsys thermalservice显示HAL Ready=false。
```

证据：

```text
thermalservice无cached temperatures；
但/sys/class/thermal下仍有battery、pmic、tsens、cooling_device。
```

结论：

```text
Framework thermal链路不可用时，热分析需要下钻Kernel thermal sysfs。
不能因为thermalservice为空就判断没有温控。
```

## Case：长时间partial wakelock

证据链：

```text
dumpsys power显示PARTIAL_WAKE_LOCK；
BatteryStats记录UID wakelock时间；
wakeup_sources中PowerManagerService.WakeLocks total_time增长；
Perfetto显示持锁期间线程运行。
```

结论：

```text
应用或系统服务持锁阻止suspend。
```

## Case：周期wakeup alarm

证据链：

```text
BatteryStats中wakeup alarm次数高；
wakeup_sources中alarmtimer增长；
Perfetto显示周期resume；
dumpsys alarm定位包名。
```

修复：

```text
改用inexact alarm或JobScheduler，合并任务并做退避。
```

## Case：亮屏静置耗电

证据链：

```text
固定亮度和刷新率后电流仍高；
Perfetto显示SurfaceFlinger/RenderThread持续运行；
CPU/GPU频率无法下降。
```

结论：

```text
持续刷新导致显示合成和计算链路活跃。
```

## Case：充电发热限流

证据链：

```text
battery temp上升；
current_now下降；
thermal_zone温度达到阈值；
cooling_device状态变化。
```

结论：

```text
充电热和运行负载叠加触发thermal限流。
```

## 简历表达

```text
建立Android功耗与热问题分析流程，结合dumpsys power、
BatteryStats、Perfetto、kernel wakeup_sources、thermal_zone和电流仪，
定位待机唤醒、wakelock、亮屏持续刷新、充电发热限流等问题。
```

## 面试表达

```text
我分析功耗不会只看一个工具。通常先确认测试条件，
再用BatteryStats找长周期归因，用Perfetto看时间线，
用wakeup_sources确认Kernel唤醒，用thermal_zone/cooling_device
确认热限制，最后用同条件电流或温度数据验证修复收益。
```

这句话比“我熟悉 PMS 和 Doze”更像真实做过问题的人。
