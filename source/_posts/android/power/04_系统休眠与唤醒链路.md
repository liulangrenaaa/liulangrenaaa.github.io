# 系统休眠与唤醒链路

在 Android 手机中，当用户按下电源键熄屏，或者手机静置超时后，系统会启动一系列动作，最终让整个芯片（SoC）进入极低功耗的休眠状态（通常是 Suspend to RAM / s2idle）。一旦有外界输入（如来电、闹钟、按键），SoC 又会被瞬间唤醒。

本节我们将基于 **AOSP 14**，打通从 Java 层的 PMS、Native 层的守护进程 `system_suspend`，直到 Linux 内核的休眠与唤醒完整链路。

---

## 一、 系统挂起（Suspend）的起点：从 Java 到 Native

在前两章中，我们知道当系统无任何 WakeLock 且屏幕关闭时，PMS 会释放 `SuspendBlocker`：

### 1. `SuspendBlocker` 的底层是 Binder 通信
在 AOSP 14 中，`SuspendBlocker` 的实现类是 `SuspendBlockerImpl`。它并不是直接写入 sysfs，而是通过 Binder 跨进程调用本地守护进程 `system_suspend`：

```java
// base/services/core/java/com/android/server/power/PowerManagerService.java

private final class SuspendBlockerImpl implements SuspendBlocker {
    private final String mName;
    private int mReferenceCount;

    @Override
    public void release() {
        synchronized (this) {
            mReferenceCount--;
            if (mReferenceCount == 0) {
                // 通过 JNI 调用 Native 层，最终释放 system_suspend 的 WakeLock
                nativeReleaseSuspendBlocker(mName);
            }
        }
    }
}
```

### 2. 本地挂起服务：`system_suspend` 守护进程
在 Android 10 及以上版本中，Google 引入了名为 **`system_suspend`** 的 Native 守护进程（源码位于 `system/hardware/interfaces/suspend/`）。
* 它实现了 `android.system.suspend` AIDL 接口。
* 它负责统一接管用户空间所有的唤醒锁请求。
* 当 `system_suspend` 检测到用户空间**所有的锁**都被释放后，它会向内核的 `/sys/power/state` 或 `/sys/power/autosleep` 写入 `mem`，触发 Linux 内核的挂起流程。

---

## 二、 内核挂起核心流程：Linux PM Core

当内核接收到写入 `/sys/power/state` 的 `mem` 指令后，Linux 内核的电源管理核心（Linux PM Core）开始接管，其流程图如下：

```
          用户空间写入 "mem" 到 /sys/power/state
                          │
                          ▼
             内核入口: state_store()
                          │
                          ▼
                enter_state(PM_SUSPEND_MEM)
                          │
                          ▼
1. 挂起准备阶段: suspend_prepare()
   ├── 广播通知各驱动准备挂起 (pm_notifier_call_chain)
   └── 冻结用户空间进程和内核线程 (freeze_processes(), freeze_kernel_threads())
                          │
                          ▼
2. 设备挂起阶段: suspend_devices_and_enter()
   ├── 暂停所有设备驱动 (dpm_suspend_start()) -> 依次调用各 driver 的 suspend()
   ├── 关停除 Boot CPU 外的所有 CPU 核心 (disable_nonboot_cpus())
   └── 暂停系统时钟 (timekeeping_suspend())
                          │
                          ▼
3. 芯片级挂起: suspend_ops->enter()
   └── 触发 SoC 平台特定的低功耗模式指令 (如 ARM 的 WFI - Wait For Interrupt)
       此时 DRAM 进入自刷新模式，SoC 停止供电，系统电流降至 2~5mA
```

### 核心阶段详解：
1. **冻结进程（Freezing Processes）**：
   为了防止在系统挂起过程中，应用程序修改磁盘文件或修改硬件状态，系统必须先将所有用户空间进程以及大部分内核线程“冻结”（处于不可中断的休眠状态，不接受任何 CPU 调度）。
2. **设备挂起（Device Suspend）**：
   内核依次遍历系统中的设备驱动树（Device Tree），调用每一个硬件驱动注册的 `suspend` 函数。例如：
   * 屏幕显示驱动关闭背光并断电。
   * Wi-Fi 芯片进入低功耗监听模式。
   * 传感器控制器关闭高频采样。
3. **进入 WFI 状态**：
   最后，主 CPU 关闭所有中断屏蔽，执行 ARM 的 `WFI` 汇编指令，芯片硬件彻底停摆，只留下硬件唤醒源（Wakeup Sources）处于监听状态。

---

## 三、 内核唤醒核心流程：Resume

当外界发生中断（例如用户按下了电源键，或者电池管理芯片产生了一个充电中断），SoC 硬件会自动恢复对主 CPU 的供电，主 CPU 从挂起时的下一行汇编指令开始执行，进入 **唤醒（Resume）** 流程：

```
               硬件触发 Wakeup Interrupt (如 Power Key 中断)
                                     │
                                     ▼
                      CPU 恢复供电，跳出 WFI 汇编
                                     │
                                     ▼
                触发底层平台唤醒: suspend_ops->leave()
                                     │
                                     ▼
                      系统时钟恢复 (timekeeping_resume())
                                     │
                                     ▼
                启动非 Boot CPU 核心 (enable_nonboot_cpus())
                                     │
                                     ▼
               恢复所有设备驱动: dpm_resume_end()
               依次调用各 driver 注册的 resume() 方法
                                     │
                                     ▼
               解冻进程: thaw_processes(), thaw_kernel_threads()
                                     │
                                     ▼
           挂起服务 `system_suspend` 重新开始工作，系统回到正常运行态
```

---

## 四、 唤醒源与不休眠排查：`wakeup_sources`

在实际功耗排查中，经常会遇到“屏幕灭屏后，手机依然无法进入休眠”或者“手机休眠后，在一分钟内被频繁唤醒了上百次（即频繁 Suspend/Resume 循环）”。

这类问题通常由**内核级唤醒源（Kernel Wakeup Sources）**异常触发导致。

### 1. 什么是 Wakeup Source？
Linux 内核中，硬件驱动为了防止系统在自己处理关键数据时挂起，会申请一个 `wakeup_source`。
例如，当 Wi-Fi 正在接收网络包时，Wi-Fi 驱动持有 `wlan` 唤醒源，这会阻止 SoC 进入 `WFI` 状态。一旦包处理完毕，驱动释放该唤醒源。

### 2. 如何查看内核当前的唤醒源？
我们可以通过读取内核节点来获取当前系统中所有唤醒源的持有状态和唤醒次数：
* 内核 4.x/5.x 之后路径：`/sys/kernel/debug/wakeup_sources`
* 也可以在已 Root 手机上使用：`cat /d/wakeup_sources`

输出的关键列包括：
* **`name`**：唤醒源的名称（如 `qpnp_pon` 对应电源键，`alarmtimer` 对应定时器，`wlan` 对应 Wi-Fi）。
* **`active_count`**：此唤醒源自开机以来激活的次数。
* **`total_time`**：阻止系统挂起的总时长。
* **`prevent_suspend_time`**：当前正在阻止系统休眠的时间。

如果发现某个唤醒源的 `prevent_suspend_time` 异常庞大，或者 `active_count` 在短时间内暴增，即可直接定位到对应的硬件驱动或外设存在故障。
