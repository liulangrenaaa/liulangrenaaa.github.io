# 网络定位蓝牙传感器功耗场景

除了屏幕和 CPU，手机中搭载的各类无线通信射频芯片（蜂窝网络、Wi-Fi、蓝牙、GPS）以及传感器也是不容小觑的耗电来源。如果后台应用频繁拉起这些硬件，会给整机带来巨大的续航压力。

本节我们将深度解析网络“尾部状态”、高精度 GPS 搜星耗电、蓝牙低功耗（BLE）扫描开销，以及传感器批处理（FIFO Batching）与 Sensor Hub 优化设计。

---

## 一、 无线网络与“尾部状态”功耗模型 (Radio Tail State)

无线蜂窝射频芯片（4G LTE / 5G NR）在工作时采用了一种特殊的**状态机调度模型**。当 App 发送一个仅有 1KB 大小的数据包时，射频芯片并不会在发完包后立刻休眠，而是会留下一段极长且极高耗电的**“尾部时间”（Tail Time）**。

```
       射频功耗 (mA)
         ▲
  150mA ─┼───[ RRC_CONNECTED (数据激活传输中) ]
         │   ┌───────────────────────────┐
   80mA ─┼───│                           │───[ RRC_INACTIVE / RRC_FACH (尾部保留状态) ]
         │   │                           │   ┌───────────────────────────┐
         │   │                           │   │                           │───[ RRC_IDLE (休眠状态) ]
    3mA ─┼───│                           │   │                           │   ┌──────
         └───┴───────────────────────────┴───┴───────────────────────────┴───┴──────────> 时间 (秒)
             |<─────── 数据传输 ───────>||<───────── 尾部时间 (5~15秒) ───>|
```

### 1. 为什么存在尾部状态？
由于射频网络从休眠状态（RRC_IDLE）重新建立与基站的连接（RRC_CONNECTED）需要进行复杂的基带握手（耗时 1~2 秒），频繁跳变会导致严重延迟。因此，移动网络协议规定：**在数据发送完毕后，射频芯片必须保持在半激活状态（RRC_INACTIVE / RRC_FACH）等待 5 至 15 秒**，以便有后续数据包时可以零延迟发出。

### 2. 尾部状态的功耗毒瘤
如果一个应用采用“每隔 15 秒发送一个心跳网络包”的野蛮设计，**整颗射频芯片将永远被困在 80mA 以上的高能耗尾部状态**，而无法回到 3mA 的 IDLE 待机态。
* **治理方案**：对网络数据进行强制批处理，拉长心跳时间。将所有后台网络上报打包，在 1 秒内发完，随后进入长深睡。

## 二、 定位与 GPS 功耗治理

高精度卫星定位（GPS / GNSS）是硬件功耗中最暴力的部分。GPS 芯片为了完成星历同步，在冷启动搜星时需要源源不断地进行射频搜索，**工作电流高达 80 ~ 150mA**。

### 1. 定位劫持与高频轮询
如果后台应用不断请求高精度的 `GPS_PROVIDER`，GPS 芯片将常亮，电池将以极快速度被抽干。

### 2. Android 14 针对定位的功耗压制
* **Fused Location Provider (FLP - 融合定位)**：
  AOSP 强烈建议使用 FLP，它在底层由 `FusedLocationProviderService` 驱动。FLP 会结合 Wi-Fi 接入点（AP）指纹、蜂窝基站 ID 以及加速度传感器，在室内或非必要场景下输出低精度定位，**完全不启动高耗电的 GPS 芯片**，使功耗降低 90% 以上。
* **后台定位节流（Background Location Throttling）**：
  在 AOSP 14 中，`LocationManagerService` 对处于后台且非前台服务的应用实施强力节流。无论应用申请多高的请求频率，**系统强行将后台定位回调限制在每小时仅触发数次（如一次或四次）**。

## 三、 蓝牙低功耗 (BLE) 扫描与硬件过滤器 (Scan Filter)

蓝牙低功耗（BLE）虽然声称“低功耗”，但在**频繁扫描（Scanning）**场景下依然是耗电黑洞。

### 1. 扫描对 CPU 的唤醒链
如果一个应用在后台执行无过滤器的 BLE 扫描：
* 周边所有蓝牙设备广播的信标数据包（Advertisements）都会被蓝牙射频芯片接收。
* 芯片收到包后触发硬件中断，强行唤醒主 CPU 运行 Android 蓝牙栈。
* 蓝牙服务再通过 Binder 回调给 App，CPU 无法入睡。

### 2. 硬件过滤（Offloaded Scan Filtering）
为了优化这一场景，现代 Android 强制推行 **硬件过滤器（Hardware Scan Filter）**。
* App 在发起扫描时，必须传入 `ScanFilter`（例如指定过滤特定设备的 MAC 地址或特定服务 UUID）。
* 蓝牙服务会将这些过滤条件直接**下发到蓝牙控制器芯片的内置内存**中。
* 只有当周边出现完全匹配过滤器的设备时，蓝牙芯片才会产生中断唤醒主 CPU；其余杂乱的广播包全部在芯片硬件层直接丢弃，主 CPU 保持完全休眠。

## 四、 传感器批处理（FIFO Batching）与 Sensor Hub

手机中的加速度计、陀螺仪等传感器，在计步器、运动追踪应用中需要以 50Hz 或 100Hz 的频率高频采集数据。

### 1. 传统的 AP 唤醒陷阱
如果每次传感器芯片采集到一个 3 轴数据，就立刻触发一次中断派发给 CPU：
* CPU 每秒要被唤醒 100 次（每 10 毫秒一次）。
* 主 SOC 完全处于“浅睡-苏醒”高频跳变，待机续航崩盘。

### 2. 硬件 FIFO 批处理 (Batching)
为了解决这一矛盾，Android 引入了 **传感器批处理（Sensor Batching）**。
* 传感器硬件内部集成了小容量的 **FIFO（先进先出）硬件缓冲区**。
* App 在注册传感器监听时，通过设置 `maxReportLatencyUs` 参数，允许系统延迟交付数据。
* 传感器采集的数据先暂存在 FIFO 缓冲区中，主 CPU 彻底休眠。
* 只有当 FIFO 缓冲区写满、或者到达最大延迟时间（如 10 秒）时，传感器才会发送一个单次中断，一次性将这 1000 个点的数据打包派发给 CPU 处理，**唤醒频率降低 1000 倍**。

```
【 传统模式 (高频唤醒) 】
  Sensor ──(每 10ms 触发中断)──> 主 CPU 苏醒 (CPU 永远无法休眠，耗电极高)
 
【 批处理模式 (FIFO Batching) 】
  Sensor ──> [ 硬件 FIFO 暂存 ] (主 CPU 保持休眠 10 秒)
                    │ (FIFO 存满或时间到)
                    └───(单次中断一次性打包投递)───> 主 CPU 一次性处理 (极省电)
```

### 3. 低功耗协同处理器 (Sensor Hub)
现代旗舰手机更引入了 **`Sensor Hub`（低功耗安全微控制器/DSP）**。
* 计步算法（Step Detector）、手势唤醒检测（Significant Motion）等，直接运行在这个只有数毫瓦功耗的 Sensor Hub 上。
* 主 SoC 保持彻底的 `Deep Suspend`。只有当 Sensor Hub 确信用户正在跑步、或者拿起了手机亮屏时，才通过特殊的唤醒中断拉起主 SoC 的 Java PMS。
