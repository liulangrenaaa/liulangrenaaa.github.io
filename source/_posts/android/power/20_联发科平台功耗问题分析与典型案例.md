# 联发科平台功耗问题分析与典型案例

联发科（MediaTek, 简称 MTK）天玑系列及芯片平台，在智能手机、平板和车载智能座舱中有着极其广泛的应用。MTK 平台在底层的低功耗控制（Low Power Control）有一套独特的技术架构体系，其核心决策完全由专用的硬件协处理器 **SPM (System Power Manager)** 驱动。

本节我们将详解 MTK 平台的低功耗状态机、MobileLog 功耗定位法，以及一个因为外设时钟未释放导致 SPM 挂起失败的经典功耗案例。

---

## 一、 MTK 平台功耗核心：SPM 与 SODI 机制

在 MTK 平台中，所有的电源状态跳转（包括睡眠、深度闲置、唤醒）都由 **SPM (System Power Manager)** 硬件微控制器全权接管。

### 1. SPM 负责的三大功耗层次
* **`Deep Idle` (深度闲置模式)**：
  * 当屏幕亮着但系统没有繁重任务时，系统会尝试让 CPU 核心关闭，并关断系统总线时钟。
* **`SODI` (Screen-On Deep Idle，亮屏深度闲置)**：
  * **MTK 独家省电神技**。当手机处于 AOD 息屏显示、或者播放静态画面（屏幕背光亮着但画面静止）时，SPM 会强制将应用处理器（AP）相关的电源域完全断电，仅留显示控制器刷新屏幕。
* **`Suspend` (系统挂起)**：
  * 灭屏待机状态。SPM 会执行一套复杂的“微码”（Microcode）动作，彻底切断 DDR 内存以外所有芯片的外设电源，让功耗沉入最深谷底。

### 2. MTK 功耗分析核心虚拟节点
在进行调试时，可以在已 Root 的 MTK 设备上查询以下节点以确认 SPM 的运转情况：
* **`/sys/kernel/debug/spm/suspend_stats`** 或 **`/sys/power/spm/suspend_stats`**：
  * **用途**：显示 SPM 挂起和唤醒的统计状态。
  * **排查关键**：如果 `suspend_scan_cnt`（挂起尝试次数）很高，但 `suspend_allow_cnt`（成功挂起数）很低，说明 SPM 被频繁打断，挂起失败率高。
* **`/sys/kernel/debug/spm/spm_state`**：
  * 实时显示当前 SPM 硬件控制器的内部状态编码（SPM State Machine Code）。

## 二、 黄金工具链：MTK MobileLog 分析法

在 MTK 平台上，最强大、最高效的分析手段莫过于抓取并解析 **`MobileLog`**。

### 1. 如何抓取 MobileLog
1. 在拨号盘输入 **`*#*#3646633#*#*`** 进入 MTK 工程模式（Engineer Mode）。
2. 找到 `Log and Debugging` -> `MTKLogger`，开启 `MobileLog`。
3. 复现待机高耗电问题，随后导出 `/sdcard/mtklog/mobilelog/` 文件夹。

### 2. MobileLog 功耗排查三板斧
MobileLog 导出的日志中，包含了极为丰富的系统级诊断信息，是高通平台所不具备的：
* **`syslog` (系统内核与服务日志)**：
  * 搜索 **`[SPM]`**：可以直观看到 SPM 的休眠失败日志、挂起被谁打断（Wakeup Source）。
  * 搜索 **`active wake lock`**：查看内核中阻止挂起的锁。
* **`fg_log` / `bat_thread` (电池库仑计日志)**：
  * MTK 平台的高精度电量计（Fuel Gauge）会以秒级在日志中输出实时电压（V）、电流（I）、温度和电量百分比，直接可以用 Excel 绘制极其精准的功耗波动折线图。

## 三、 典型案例：I2C 外设时钟（INFRA_CLK）未释放导致 SPM 无法 Suspend 案

### 1. 问题现象
* **测试报告**：某款搭载天玑 9200 芯片的 MTK 平板电脑，在纯待机测试下，灭屏电流死锁在 **12mA** 左右（正常基线应小于 **1.5mA**）。
* **初步排查**：检查 Java 层的 PMS 与 Android 限制均正常，但平板外壳有微弱温热感。

### 2. MobileLog 分析与 SPM 节点定位
1. **抓取 MobileLog**：
   开启 MobileLog，进行灭屏静置测试 15 分钟，随后将日志导出。
2. **分析 syslog 中的 SPM 日志**：
   在 `syslog` 中过滤关键字 `[SPM]`，发现了海量如下警报：
   ```log
   [SPM] SPM suspend aborted! wdt_val = 0x00010000, reason = 0x4 (INFRA_CLK_ACTIVE)
   [SPM] SPM cannot enter Deep Sleep because INFRA_CLK is locked by peripheral!
   ```
   * **原因解析**：SPM 挂起逻辑报错，指出系统核心的内部外设总线时钟 **`INFRA_CLK` 依然处于 Active 激活状态**。按照微码安全规范，若总线时钟未关闭，SPM 强制切断电源会导致总线硬件损坏，因此 SPM 挂起被安全中止（Aborted）。

3. **追查时钟锁持有者**：
   过滤时钟树变化日志，追查是哪个外设驱动在灭屏后依然拽着 `INFRA_CLK` 不放。
   * 发现系统时钟树中，`INFRA_I2C3` 时钟的引用计数（Reference Count）始终为 `1`。
   * `I2C3` 总线挂载的物理器件是：**触控手写笔的充电控制芯片（Stylus Charger IC）**。

### 3. 根因剖析
* 调阅触控笔充电 IC 的驱动源码：
  发现该驱动注册的 Linux PM 挂起回调函数 `stylus_charger_suspend` 存在代码缺陷。
* 当平板熄屏时，驱动会向手写笔发起一次 I2C 握手查询电量。
* 如果此时手写笔没有吸附在平板上（处于断开状态），I2C 传输会产生超时报错（Timeout）。
* 驱动在捕获到 I2C 传输错误后，**忘记调用 `clk_disable_unprepare()` 释放总线时钟**，导致 `INFRA_CLK` 的引用计数被永久吊死在 `1`，进而导致整机的 SPM 控制器无法完成深睡微码动作。

### 4. 解决方案
修改充电芯片的内核驱动代码，在 I2C 报错的异常分支中，加入完整的时钟释放与复位逻辑：

```c
// 驱动修复片段
static int stylus_charger_suspend(struct device *dev) {
    int ret;
    ret = query_stylus_status();
    if (ret < 0) {
        pr_err("Query stylus failed during suspend\n");
        // 修复：确保在发生错误时，依然关闭并释放 I2C 外设时钟
        clk_disable_unprepare(charger_data->infra_i2c_clk);
        return ret;
    }
    clk_disable_unprepare(charger_data->infra_i2c_clk);
    return 0;
}
```

* **效果验证**：重新编译内核并刷机测试，灭屏后 MobileLog 日志显示：
  `[SPM] suspend successfully! enter DEEP_SLEEP mode.`
  系统成功进入休眠，整机待机底流下降至 **1.2mA**，问题顺利解决！
