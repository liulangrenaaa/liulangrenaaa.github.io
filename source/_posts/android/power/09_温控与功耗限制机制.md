# 温控与功耗限制机制

随着现代手机芯片性能的飙升，当 CPU、GPU 满负载工作，或者手机处于大功率快充时，电能转化为热能，整机发热量会剧烈增加。如果不加控制，不仅会引起用户手感发烫，甚至会导致锂电池发生危险或硬件永久损坏。

为了实现控温，Android 引入了 **`ThermalManagerService` (TMS)**，建立了一套精细化的温控调节与功耗限制（Thermal Throttling）机制。

本节我们将基于 **AOSP 14**，详解 Android 的温控体系设计与动态缓解策略。

---

## 一、 Android 温控架构设计

Android 的温控系统（Thermal Framework）同样采用了从内核传感器到 Java 框架的垂直架构：

```
┌────────────────────────────────────────────────────────────────────────┐
│ 1. 应用程序与核心服务 (Apps & Services)                                  │
│    - 监听 Thermal Status 广播或回调，主动降低自身负载 (如降低录制分辨率)    │
│    - DisplayPowerController (根据温控等级强制压低背光)                   │
└───────────────────────────────────┬────────────────────────────────────┘
                                    │ Binder IPC
┌───────────────────────────────────▼────────────────────────────────────┐
│ 2. 核心服务层 (ThermalManagerService - TMS)                            │
│    - 监听 Thermal HAL 的温度事件                                       │
│    - 计算并发布系统的温控等级 (THERMAL_STATUS_*)                        │
└───────────────────────────────────┬────────────────────────────────────┘
                                    │ AIDL (android.hardware.thermal)
┌───────────────────────────────────▼────────────────────────────────────┐
│ 3. 硬件抽象层 (Thermal HAL - IThermal.aidl)                             │
│    - 提供温度传感器类型 (CPU, GPU, Battery, SKIN)                       │
│    - 收集芯片内部温度以及外壳皮肤 (SKIN) 温度                            │
└───────────────────────────────────┬────────────────────────────────────┘
                                    │ Hwmon Sysfs / Driver API
┌───────────────────────────────────▼────────────────────────────────────┐
│ 4. 内核层 (Kernel Thermal Governor)                                    │
│    - Linux 内核 IPA (Intelligent Power Allocation) 智能功耗分配算法      │
│    - 硬件温控驱动直接通过 cpufreq 强行限制 CPU 频率                      │
└────────────────────────────────────────────────────────────────────────┘
```

## 二、 核心温控状态机：7 个发热等级

在 AOSP 14 中，`ThermalManager` 中定义了 7 个温控严重等级（Thermal Status）。系统各个子模块会根据当前处于哪个等级，采取不同层级的“功耗熔断”措施：

| 温控状态等级 | 常规外壳温度 | 系统执行的功耗限制与热降退策略 |
| :--- | :--- | :--- |
| **`THERMAL_STATUS_NONE` (0)** | $< 37^\circ C$ | **正常状态**。无任何温控限制，系统可以满载运行。 |
| **`THERMAL_STATUS_LIGHT` (1)** | $\approx 37 \sim 39^\circ C$ | **轻微发热**。后台开始限制部分非紧急 Job，稍许调低大核最大频率。 |
| **`THERMAL_STATUS_MODERATE` (2)**| $\approx 40 \sim 42^\circ C$ | **中度发热**。CPU 大核开始限频，GPU 适当降频，屏幕最大亮度降低。 |
| **`THERMAL_STATUS_SEVERE` (3)** | $\approx 43 \sim 45^\circ C$ | **严重发热**。屏幕刷新率强行**从 120Hz 降至 60Hz**，背光强制压低，相机可能停用高帧率录制，关闭闪光灯。 |
| **`THERMAL_STATUS_CRITICAL` (4)**| $\approx 46 \sim 48^\circ C$ | **临界发热**。CPU/GPU 深度限频（大核几乎停摆，只留小核工作），蜂窝网络进入低功耗监听模式，手机开始出现严重卡顿。 |
| **`THERMAL_STATUS_EMERGENCY` (5)**| $\approx 49 \sim 51^\circ C$ | **紧急发热**。除紧急通话外，所有网络连接断开，部分非核心系统服务直接杀掉，停止一切多媒体播放与录制。 |
| **`THERMAL_STATUS_SHUTDOWN` (6)** | $> 52^\circ C$ | **强行关机**。为了防止主板烧毁或电池爆炸，系统直接调用安全关机指令。 |

## 三、 外壳皮肤温度（SKIN）是核心参考标准

在早期的手机中，温控系统主要读取 CPU 芯片内的温度传感器（Junction Temperature）。但这会导致一个极差的体验：**芯片耐热极高（可达 $105^\circ C$），当芯片 $80^\circ C$ 时，手机外壳已经达到了 $50^\circ C$，用户手掌早已被烫伤。**

为此，现代 Android 设备的温控模型高度依赖 **`SKIN`（外壳/皮肤）传感器**。
* 手机厂商会在主板最靠近外壳的物理位置布置数个高精度 NTC 热敏电阻。
* Thermal HAL 将其归类为 `TemperatureType.SKIN`。
* 所有的温控降退策略，主要以 SKIN 传感器的温度为基准，确保用户接触到的手感温度绝对不超过安全临界值（如 $45^\circ C$）。

## 四、 动态热缓解（Thermal Mitigation）实战手段

一旦 `ThermalManagerService` 计算出温控等级升级，它会通过 Binder 接口向全系统注册了监听的回调（`IThermalStatusListener`）派发通知，触发**多维度立体式功耗大压制**：

### 1. 屏幕降亮与降帧
* **`DisplayPowerController`** 收到 Severe 等级后，立即启用温控降亮曲线：
  $$\text{最大允许背光} = \min(\text{用户设定背光}, \text{温控最大背光限额})$$
* 屏幕显示芯片调用接口，强行锁定 60Hz（甚至 30Hz）刷新率，大幅降低 GPU 和面板发热。

### 2. CPU / GPU 降频与 EAS 能量模型限制
* **`Power HAL`** 接收到降级通知后，会修改 CPU 调频策略，不再响应应用的 “Boost” 提频请求。
* 内核的 **`EAS`** 调度器会主动把大任务从超大核（Cortex-X系列）或大核驱逐，强制分摊到发热量极低的小核（Cortex-A系列）上运行。

### 3. 后台进程深度限制（冻结）
* **`ActivityManagerService`** 限制处于 Cached 状态的后台进程，通过 `cgroups` 彻底剥夺其 CPU 时间片，防止后台进程“捣乱”产生额外热量。

## 五、 小结与常用调试命令

温控是系统流畅度与安全续航的博弈。在测试和研发中，我们可以通过以下命令实时查询和模拟温控状态：

* **查询当前温控服务状态与传感器实时温度**：
  `adb shell dumpsys thermalservice`
* **模拟强行切换整机温控等级到 SEVERE (3)**：
  `adb shell cmd thermalservice override-status 3`
