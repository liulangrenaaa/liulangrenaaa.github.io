# 唤醒锁机制详解

在 Android 系统中，当屏幕熄灭（Screen Off）后，系统默认会迅速尝试让整个 SOC 进入休眠状态（Deep Suspend）以节省电量。为了使应用能在后台继续运行（例如：后台下载、播放音乐、同步数据），Android 引入了 **唤醒锁（WakeLock）** 机制。

本节我们将深度解析 WakeLock 的底层设计、核心类型、生命周期治理以及 AOSP 14 中负责电量归因的 `WorkSource` 机制。

---

## 一、 WakeLock 的核心类型与等级

在应用层，WakeLock 主要通过 `PowerManager` 创建。根据作用范围和生命周期，主要分为以下几类：

### 1. 核心锁等级 (Lock Levels)
* **`PARTIAL_WAKE_LOCK`**：
  * **最常用、也是最容易导致耗电问题的锁**。
  * **表现**：屏幕熄灭后，**CPU 保持运转**，不会进入休眠状态。
* **`PROXIMITY_SCREEN_OFF_WAKE_LOCK`**：
  * **表现**：配合接近传感器（Proximity Sensor）使用。当电话贴近耳朵时，屏幕自动熄灭；拿开时，屏幕重新亮起。常用于通话软件。
* **`DOZE_WAKE_LOCK` & `DRAW_WAKE_LOCK`**：
  * 系统级特殊的锁。主要由 SystemUI 或壁纸组件持有，用于在 Doze 状态下临时亮屏绘制界面。

> [!WARNING]
> 以下三个等级在现代 Android 版本中均已被**废弃**（Deprecated）。如果需要在应用内保持屏幕常亮，推荐在 Activity 的 Window 中添加 `WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON` 标志，该标志在 Activity 销毁时会自动释放，不易发生泄露：
> * `SCREEN_DIM_WAKE_LOCK`
> * `SCREEN_BRIGHT_WAKE_LOCK`
> * `FULL_WAKE_LOCK`

## 二、 PMS 内部 WakeLock 数据结构 (AOSP 14)

在 PMS 内部，每把 WakeLock 都是一个 `PowerManagerService.WakeLock` 对象，核心成员变量如下：

```java
// base/services/core/java/com/android/server/power/PowerManagerService.java

public final class WakeLock implements IBinder.DeathRecipient {
    public final IBinder mToken;       // 客户端申请锁持有的 Binder 令牌
    public final int mFlags;            // 锁的类型标志 (如 PARTIAL_WAKE_LOCK)
    public final String mTag;           // 开发者传入的锁名称标签，调试定位核心
    public final WorkSource mWorkSource;// 锁电量归因的 WorkSource
    public final int mOwnerUid;         // 申请锁的应用 UID
    public final int mOwnerPid;         // 申请锁的应用 PID
    ...
}
```

### 1. 死亡代理机制（DeathRecipient）
为什么 WakeLock 要实现 `IBinder.DeathRecipient` 接口？
**为了防止锁泄露**。当一个应用申请了 Partial WakeLock 后，如果在释放前进程崩溃（Crash）或被系统杀掉（Kill），这把锁就会一直残留在 `system_server` 中，导致系统永远无法休眠。
PMS 通过绑定 Binder 的死亡监听器（`mToken.linkToDeath(this, 0)`），在进程意外死亡时，自动触发锁的释放逻辑：

```java
public void binderDied() {
    synchronized (mLock) {
        handleWakeLockDeath(this); // 进程死亡，PMS 自动释放该锁
    }
}
```

## 三、 WakeLock 申请（Acquire）与释放（Release）的核心链路

### 1. 锁申请（Acquire）流程图

```
App.java (PowerManager.WakeLock.acquire())
  │ 
  ▼  Binder IPC (调用 IPowerManager.aidl)
PowerManagerService.java (acquireWakeLock())
  │ 
  ▼  权限检查 (如 android.permission.WAKE_LOCK)
PowerManagerService.java (acquireWakeLockInternal())
  │ 
  ├── 1. 创建新 WakeLock 并 linkToDeath 死亡代理
  ├── 2. 将锁加入 PMS 的全局 List mWakeLocks 中
  ├── 3. 调用 mNotifier.onWakeLockAcquired() (通知 BatteryStats 开始记账)
  └── 4. 调用 updatePowerStateLocked() (通知核心电源状态机更新)
```

### 2. 锁释放（Release）流程
锁释放流程是申请的逆过程：
1. 客户端通过 Binder 调用 PMS 的 `releaseWakeLock()`。
2. PMS 从 `mWakeLocks` 列表中移除对应的 `WakeLock` 对象，并解除死亡代理（`unlinkToDeath`）。
3. 调用 `mNotifier.onWakeLockReleased()`（通知 BatteryStats 停止记账）。
4. 调用 `updatePowerStateLocked()` 重新计算 `mWakeLockSummary`，如果全系统无锁，则释放底层的 `SuspendBlocker`。

## 四、 电量归因机制：`WorkSource`

在实际开发中，经常会有**系统服务或公共下载器代应用干活**的场景。例如：系统 `DownloadManager` 代应用 A 下载一个 1GB 的文件，`DownloadManager` 申请了 Partial WakeLock。
如果直接统计，这 1GB 产生的功耗会被记在 `DownloadManager`（系统 UID）头上，导致耗电排行不准。

为了解决这一问题，Android 引入了 **`WorkSource`** 机制。

```java
// 申请 Partial WakeLock 时携带 WorkSource
WorkSource ws = new WorkSource(targetAppUid, targetPackageName);
PowerManager.WakeLock wakeLock = powerManager.newWakeLock(
    PowerManager.PARTIAL_WAKE_LOCK, "MyDownloadService");
wakeLock.setWorkSource(ws);
wakeLock.acquire();
```

### WorkSource 工作原理
当 `BatteryStatsService` 统计电量时，它会识别 PMS 传入的 `WorkSource`：
* 它**不会**把电量记给申请锁的调用者进程（`DownloadManager`）。
* 而是将这段时间的 CPU 活跃度、功耗，**精确地归因到 `targetAppUid`（应用 A）**。
这保证了即使在复杂的协同工作下，Android 的电量排行也能做到百分百公平公正。

## 五、 WakeLock 常见耗电异常与治理手段

在实际的功耗调优中，关于 WakeLock 的异常主要集中在两点：**锁泄露** 与 **长时间持有**。

### 1. 锁泄露（Leak）
**原因**：由于代码逻辑缺陷，如在 `try-catch` 中发生异常、或者 Activity 生命周期的不当匹配（如在 `onResume` 中 acquire，但在 `onPause` 中忘记 release）。
**解决策略**：
* 强烈建议在使用 `acquire` 时，一律强制带上**超时释放**机制：
  `wakeLock.acquire(10 * 60 * 1000L); // 10分钟后无论如何强制释放`

### 2. 长时间持有（Long-held WakeLock）
**原因**：应用虽然正常释放了锁，但单次持有时间太长（如后台持续上传、下载或播放音乐）。
**系统级治理手段 (Android 14 防御机制)**：
* **`BatteryStats` 警报**：如果一个非白名单应用持有 Partial WakeLock 超过 10 分钟，系统会在 Logcat 中输出警告，并在后台强制执行限制。
* **FGS (Foreground Service) 限制**：在 Android 14 中，后台进程不允许随意拉起前台服务，且对后台 WakeLock 的限制更加严苛。若进程进入 Cached 状态，其持有的所有 WakeLock 会被 PMS 强行“冻结”（Disabled），停止阻止内核休眠。
