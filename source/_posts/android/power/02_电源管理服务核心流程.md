# 电源管理服务核心流程

`PowerManagerService`（简称 PMS）是 Android Framework 中负责电源管理的核心系统服务。它统管着全系统的屏幕亮度、系统唤醒/休眠状态、唤醒锁（WakeLock）的申请与释放，并与 `DisplayManagerService`（DMS）以及底层的 Power HAL 紧密配合。

本节我们将基于 **AOSP 14** 源码，深度剖析 PMS 的启动流程、核心状态机变化以及电源状态更新的核心方法 `updatePowerStateLocked()`。

---

## 一、 PMS 的启动与初始化流程

PMS 继承自 `SystemService`，运行在 `system_server` 进程中。其启动与初始化主要分为两个阶段：构造与生命周期回调。

### 1. 启动入口
在 `SystemServer.java` 的 `startCoreServices()` 中，PMS 被创建并注册到 ServiceManager 中：

```java
// base/services/java/com/android/server/SystemServer.java
private void startCoreServices(@NonNull TimingsTraceAndSlog t) {
    ...
    // 创建并启动 PowerManagerService
    mPowerManagerService = mSystemServiceManager.startService(PowerManagerService.class);
    ...
}
```

### 2. 核心初始化 `onStart()` 与 `onBootPhase()`
PMS 在实例化和不同引导阶段会执行相应的初始化：

* **`onStart()`**：
  * 创建 `HandlerThread`（命名为 `PowerManagerService`），用于处理异步消息。
  * 绑定 Native 层的 Power HAL 服务（通过 JNI 调用）。
  * 注册系统的 `Binder` 接口（`IPowerManager.Stub`）。
  * 获取 Native 层的 `SuspendBlocker` 实例。

* **`onBootPhase(int phase)`**：
  在系统启动的各个关键阶段（如 `PHASE_SYSTEM_SERVICES_READY`、`PHASE_BOOT_COMPLETED`），PMS 会：
  * 绑定 `BatteryService`、`DisplayManagerService`、`ActivityManagerService` 等外部服务。
  * 注册传感器监听（例如配合自动亮度的光线传感器）。
  * 读取系统的策略配置（如 `power_profile.xml` 和 `config.xml`）。

## 二、 PMS 核心状态机：Wakefulness

PMS 内部使用 `Wakefulness` 来标识当前设备所处的电源状态。Android 14 中定义了 4 种核心 `Wakefulness` 状态：

```
                    wakeUp() (如按下电源键/收到来电)
        ┌────────────────────────────────────────────────────────┐
        │                                                        │
        ▼                                                        │
┌──────────────┐      Dreaming 结束或主动切换      ┌─────────────┴┐
│  Asleep      ├─────────────────────────────────>│ Interactive  │
│ (灭屏/深度休眠)│                                  │(亮屏/用户交互中)│
└────────┬─────┘                                  └──────┬───────┘
         ▲                                               │
         │ goToSleep() (如超时无操作/按下电源键)            │ goToSleep()
         │                                               ▼
         │                                        ┌──────────────┐
         └────────────────────────────────────────┤   Dozing     │
                       Dreaming (屏保/AOD 状态)   │(低功耗 AOD / │
                                                  │ 息屏显示状态) │
                                                  └──────────────┘
```

* **`WAKEFULNESS_ASLEEP`**：设备处于休眠状态，屏幕已关闭，CPU 准备（或已经）进入 Deep Suspend 状态。
* **`WAKEFULNESS_INTERACTIVE`**：设备处于完全交互状态，屏幕已点亮，CPU 处于活跃状态。
* **`WAKEFULNESS_DOZING`**：设备处于息屏显示（AOD）或低电量待机状态，屏幕可能处于极低功耗刷新模式，CPU 间歇性工作。
* **`WAKEFULNESS_DREAMING`**：设备正在播放屏保（Daydream），屏幕开启，但没有与用户发生直接交互。

## 三、 PMS 的心脏：`updatePowerStateLocked()` 流程分析

PMS 几乎所有的状态改变（如申请锁、释放锁、用户触摸屏幕、充电状态变化、屏幕超时等），最终都会调用 **`updatePowerStateLocked()`** 方法。这是 PMS 的核心决策中枢，它通过“多阶段流水线”计算并更新系统状态。

```
updatePowerStateLocked()
  ├── 1. updateIsPoweredLocked()              --> 计算当前充电与电池状态
  ├── 2. updateStayOnLocked()                 --> 计算是否需要保持屏幕常亮
  ├── 3. updateScreenBrightnessBoostLocked()  --> 处理亮度瞬时暴增(如双击亮屏瞬间)
  ├── 4. updateUserActivitySummaryLocked()    --> 汇总用户活动状态(超时时间计算)
  ├── 5. updateWakefulnessLocked()            --> 核心状态机决策(切换 Wakefulness)
  ├── 6. updateDisplayPowerStateLocked()      --> 投递屏幕亮度与状态给 DMS/DPC
  ├── 7. updateDreamLocked()                  --> 更新屏保(Dream)状态
  └── 8. updateSuspendBlockerLocked()         --> 计算并向内核投递 SuspendBlocker
```

以下是这 8 个阶段的核心细节：

### 阶段 1：`updateIsPoweredLocked`
检测当前是否连接充电器、USB 或是无线充电。这会影响系统的休眠策略（例如：充电时默认不休眠）。

### 阶段 2：`updateStayOnLocked`
结合开发者选项中的“充电时屏幕常亮”等配置，决定当前是否需要强制保持屏幕点亮。

### 阶段 3：`updateScreenBrightnessBoostLocked`
当用户双击电源键或者有特殊亮屏手势时，计算是否需要临时将屏幕亮度调至最大。

### 阶段 4：`updateUserActivitySummaryLocked`
计算用户活动的倒计时。根据 `mLastUserActivityTime`（最后一次用户操作时间），结合系统息屏超时设置（例如 30 秒），计算出系统当前是处于：
* `USER_ACTIVITY_SCREEN_BRIGHT`（亮屏）
* `USER_ACTIVITY_SCREEN_DIM`（暗屏）
* 还是应该休眠（Timeout）。

### 阶段 5：`updateWakefulnessLocked`
**最核心的状态转换决策点**。如果上一步计算发现用户活动已超时，且没有 WakeLock 持有，此阶段会将 `Wakefulness` 状态从 `Interactive` 切换为 `Dozing` 或 `Asleep`。

### 阶段 6：`updateDisplayPowerStateLocked`
PMS 本身不直接操作显示驱动。在这个阶段，它会将计算出的屏幕状态（亮、暗、灭）和目标亮度，封装成 `DisplayPowerRequest`，通过 `mDisplayManagerInternal.requestPowerState()` 发送给 `DisplayManagerService` 的 `DisplayPowerController`（DPC），由 DPC 真正驱动物理屏幕亮灭。

### 阶段 7：`updateDreamLocked`
如果当前状态支持，且系统配置了屏保，启动或关闭屏保（Dreaming）服务。

### 阶段 8：`updateSuspendBlockerLocked`
**打通 Java 框架与内核的终点站**。PMS 内部持有两把 Native 锁：
* **`mDecurSyncSuspendBlocker`**（同步锁）：只要屏幕亮着、或者有任何 Partial WakeLock 持有，就拉起这把锁。
* **`mDecurHalfWakeSuspendBlocker`**：配合过渡状态。

当没有 WakeLock 且屏幕已完全关闭时，PMS 会释放这两把 `SuspendBlocker`。底层 JNI 随之释放 Native 层的锁，并最终写入 `/sys/power/wake_unlock`，此时 Linux 内核开始尝试将系统挂起（Suspend to RAM）。

## 四、 核心代码片段走读 (AOSP 14)

以下是 `PowerManagerService.java` 中 `updateSuspendBlockerLocked()` 的核心逻辑简化版：

```java
// base/services/core/java/com/android/server/power/PowerManagerService.java

private void updateSuspendBlockerLocked() {
    // 1. 判断是否需要阻止内核休眠
    // 只要有任何非 Partial 锁，或者屏幕处于亮屏/Dozing状态，都必须持有物理锁
    boolean needWakeLockSuspendBlocker = ((mWakeLockSummary & WAKE_LOCK_CPU) != 0);
    boolean needDisplaySuspendBlocker = needDisplaySuspendBlockerLocked();

    // 2. 如果满足任意条件，就 acquire 同步锁，否则 release
    if (needWakeLockSuspendBlocker || needDisplaySuspendBlocker) {
        if (!mHoldingWakeLockSuspendBlocker) {
            mWakeLockSuspendBlocker.acquire(); // 最终通过 JNI 阻止 Linux 挂起
            mHoldingWakeLockSuspendBlocker = true;
        }
    } else {
        if (mHoldingWakeLockSuspendBlocker) {
            mWakeLockSuspendBlocker.release(); // 允许 Linux 挂起
            mHoldingWakeLockSuspendBlocker = false;
        }
    }
}
```

## 五、 小结与学习建议

理解 PMS 的关键在于把握 **“状态汇总 -> 状态机更新 -> 驱动下发”** 的循环闭环。

在调试功耗时，若系统无法正常休眠，我们可以使用命令：
`adb shell dumpsys power`
该命令输出的 `Wakefulness`、`Suspend Blockers` 和 `Wake Locks` 汇总，正是由 PMS 的 `updatePowerStateLocked()` 计算出来的结果。
