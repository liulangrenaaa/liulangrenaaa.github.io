# CPU 调频调度与低功耗状态

CPU 是芯片（SoC）中耗电的主力，也是计算资源的核心提供者。在 Android 设备上，CPU 的能耗曲线呈现高度非线性：**主频越高，所需的电压呈平方级上升，功耗呈立方级飙升。** 因此，如何在保障系统极致流畅的同时，最大化压低 CPU 能耗，是整机功耗控制的核心战场。

本节我们将深入内核，剖析 CPUFreq 调频机制、EAS 能量感知调度，以及 CPU 硬件的低功耗状态（C-States / P-States）。

---

## 一、 CPUFreq 变频器与 Schedutil 调频协作者

在 Linux 内核中，负责动态调整 CPU 频率以匹配当前负载的机制称为 **`CPUFreq`（CPU Frequency Scaling）**。

### 1. 传统调频器（Governors）
传统 Linux 调频器工作在定时采样模型下，常见的有：
* `performance`：强制让 CPU 锁死在最高频运行，功耗极高。
* `powersave`：强制让 CPU 锁死在最低频运行，极卡顿。
* `ondemand`：根据定时采样，当负载超过 80% 瞬时提速，负载低时降频。但在异构多核上响应滞后。

### 2. 现代 Android 标配：`schedutil` 调频器
现代 Android 设备默认采用 **`schedutil`**。它不再使用定时采样，而是**直接与进程调度器（Scheduler）进行通信**。
* **PELT (Per-Entity Load Tracking)**：调度器会以极高频率追踪每个进程的 CPU 负载占比。
* 只要一个进程被调度或苏醒，`schedutil` 就能立刻读取其 PELT 负载，并使用以下公式算出目标频率：
  $$\text{目标频率} = \text{最大主频} \times 1.25 \times \text{CPU 负载占比}$$
* 这种直接读取调度器负荷的机制，实现了**微秒级**的频率响应：当有前台滑动操作时，主频可以在 1 毫秒内瞬间飙升至峰值（避免卡顿），而滑动停止后又可以瞬间滑落到最低频（节省电量）。

## 二、 EAS 能量感知调度器 (Energy Aware Scheduling)

在 ARM 的异构多核架构（大中小核，如 1超大核 + 3大核 + 4小核）中，将一个任务分配给超大核（X核）还是小核运行，其功耗和性能差距极其悬殊。
为了解决这一问题，Android 联合 ARM 开发了 **`EAS` (Energy Aware Scheduling，能量感知调度器)**。

```
                              待调度任务 (Task)
                                      │
                       ┌──────────────┴──────────────┐
                       ▼                             ▼
              [ 方案 A: 派发给大核心 ]        [ 方案 B: 派发给小核心 ]
                       │                             │
                       ▼                             ▼
             根据 EAS 能量模型 (EM)         根据 EAS 能量模型 (EM)
             评估增加的总系统功耗 (ΔPa)       评估增加的总系统功耗 (ΔPb)
                       │                             │
                       └──────────────┬──────────────┘
                                      │ 比较 ΔPa 与 ΔPb
                                      ▼
                      选择 ΔP 最小 且 能满足 Deadline 的核心
```

### 1. 核心底盘：能量模型 (Energy Model - EM)
在系统初始化时，内核会加载一份由厂商编写的能量模型。这套模型以表格形式，明确给出了：
* 各个核心集群（Cluster 0 - 小核, Cluster 1 - 大核, Cluster 2 - 超大核）在每一个可用频率下的 **实际物理功耗（以毫瓦 mW 为单位）**。

### 2. 调度选择机制
当有一个前台任务需要被指派 CPU 时：
* EAS 会模拟把任务放进小核，以及放进大核。
* 根据能量模型，算出**方案 A 会让整机功耗增加多少毫瓦 ($\Delta Pa$)**，**方案 B 会让整机功耗增加多少毫瓦 ($\Delta Pb$)**。
* 如果小核目前的算力富余，足以在规定帧时间内（如 8.3ms）处理完该任务，EAS 会强制将任务绑定到小核运行，**哪怕大核此时完全空闲**。
* 只有当小核已经饱和、或者任务极重时，才会动用大核或超大核。这避免了“大炮打蚊子”导致的电量无谓流失。

## 三、 CPU 硬件功耗状态：P-States 与 C-States

除了在运行时调整频率，当 CPU 没有任务需要处理（处于 Idle 状态）时，硬件本身支持多种极深的睡眠等级：

```
 CPU 工作状态 (Active) ───[DVFS / P-States 变频]───> 调节频率/电压
 
                                │ 核心空闲 (Idle)
                                ▼
 ┌─────────────────────────────────────────────────────────────┐
 │ CPU 硬件睡眠状态 (C-States / CPUIdle)                         │
 │  - C0: 活跃状态 (Running)                                    │
 │  - C1: 时钟门控 (Clock Gating) ─ 关断时钟, 毫秒级恢复          │
 │  - C2: 功率门控 (Power Gating) ─ 关断供电, 需保存寄存器       │
 │  - C3/C4+: 集群级断电 ─ 整个 Cluster 彻底断电，耗时最久        │
 └─────────────────────────────────────────────────────────────┘
```

### 1. P-States (Performance States / DVFS)
* **原理**：处于活跃状态下，通过动态电压频率调整（DVFS）来实现功耗匹配。高 P-State 代表高主频和高工作电压，低 P-State 代表低主频和低电压。

### 2. C-States (CPUIdle States)
当 CPU 核心没有就绪任务时，系统会进入 `CPUIdle` 循环。内核的 `CPUIdle Governor`（如 `teo` 或 `menu` 调节器）会根据下一次定时器中断爆发的预期时间，决定将 CPU 放入哪一级 C-State：
* **`C1 (Clock Gating)`**：
  * 只关闭核心的内部时钟（Clock），但电源供电保持。
  * **特点**：功耗稍微降低，但退出和进入只需纳秒级，极其迅速。
* **`C2 (Power Gating / Core Power Off)`**：
  * 将整个核心的电源闸门（Power Gate）关闭，断电。
  * **特点**：进入前必须把核心的 CPU 寄存器状态转存到二级/三级缓存（L2/L3 Cache）。功耗极低，但唤醒时需要较长的时间重新加电和恢复寄存器。
* **`C3 及更深 (Cluster Sleep)`**：
  * 当一个 Cluster 内的所有核心全部进入 C2 时，系统会把整个 Cluster 的公共缓存断电，并把总线关断。
  * **特点**：接近静止待机功耗，但唤醒延迟最大。

## 四、 功耗调优实战指标

在进行 CPU 功耗调优时，我们核心关注两点：
1. **CPU 负载分布（Core Residency）**：使用 Perfetto 工具分析，确保在非重载场景（如看视频），90% 以上的任务都落在小核上运行，超大核的 Residency 应当趋近于 0。
2. **CPUIdle 比例**：在手机待机时，观察各 CPU 是否能迅速进入最深级别的 C-State（如 C2 或更低）。如果 CPU 频繁在 C0 和 C1 之间反复跳变（即“浅睡”），通常是因为系统存在高频的定时唤醒源（Alarm 或是底层中断）。
