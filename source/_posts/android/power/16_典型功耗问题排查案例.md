# 典型功耗问题排查案例

功耗调优是一门极度依赖“实战经验”的学问。为了让你快速掌握功耗排查的分析套路，本节为你还原 **3 个工业界最典型、高频发生的真实功耗排查案例**。我们将通过“问题现象 -> 分析步骤 -> 根因定位 -> 解决方案”的标准闭环进行层层剥茧。

## 案例一：后台 WakeLock 泄露（导致设备永久不休眠）

### 1. 问题现象
* **测试报告**：某款社交 App 在后台静置测试中，夜间纯待机（8小时）平均每小时掉电 **8.2%**（正常基线为每小时 $< 1.0\%$）。
* **初步排查**：手机屏幕完全熄灭，没有发热，但电量消耗极大。

### 2. 分析步骤与工具链
1. **抓取 bugreport**：
   重置电量后静置复现 4 小时，随后导出 `bugreport.zip`，上传至 **`Battery Historian`**。
2. **观察时间轴（Timeline）**：
   * 发现 `Screen` 轨道完全处于 Off（灭屏）。
   * 但 **`CPU running` 轨道呈现一整条连续不断的实色横线（100% 处于 Active 状态）**。
   * 这直接证实了：有服务持锁泄露，阻止了系统挂起。
3. **定位异常锁**：
   * 在 Battery Historian 底部展开 `System Stats` -> `Long-held Wakelocks`。
   * 列表中赫然排在第一名的是：
     * **Owner UID**: `10245`（对应目标社交 App）
     * **WakeLock Tag**: `"KeepAliveService:NetworkSync"`
     * **单次持有时间**: `14400 秒`（正好等于整个测试时长！）

### 3. 根因剖析（源码诊断）
开发人员定位到 App 内部的代码，发现其在进行网络同步时采用了以下逻辑：

```java
// 错误示范代码
public void syncDataInBackground() {
    PowerManager.WakeLock wakeLock = mPowerManager.newWakeLock(
        PowerManager.PARTIAL_WAKE_LOCK, "KeepAliveService:NetworkSync");
    
    wakeLock.acquire(); // 申请锁
    
    // 执行网络请求，同步数据
    mNetworkApi.doSyncRequest(); // 这是一个同步网络阻塞调用
    
    wakeLock.release(); // 释放锁
}
```
* **致命缺陷**：当网络请求 `doSyncRequest()` 因信号差或超时抛出 `IOException` 或 `RuntimeException` 时，代码运行中断并直接跳出方法。**底部的 `wakeLock.release()` 永远无法被执行到**！
* 导致这把 Partial WakeLock 永久残留在 PMS 的 `mWakeLocks` 列表中，引发严重的锁泄露。

### 4. 解决方案
重构代码，将锁释放逻辑放入 `finally` 块中，并务必加上 **超时释放** 保险机制：

```java
// 修复后代码
public void syncDataInBackground() {
    PowerManager.WakeLock wakeLock = mPowerManager.newWakeLock(
        PowerManager.PARTIAL_WAKE_LOCK, "KeepAliveService:NetworkSync");
    
    // 设定 60 秒最大超时防线，即使发生未知异常，60秒后系统也会强制释放该锁
    wakeLock.acquire(60 * 1000L); 
    
    try {
        mNetworkApi.doSyncRequest();
    } catch (Exception e) {
        Log.e(TAG, "Sync failed", e);
    } finally {
        if (wakeLock.isHeld()) {
            wakeLock.release(); // 确保安全释放
        }
    }
}
```

## 案例二：后台滥用高频精确闹钟（戳醒 CPU 无法深睡）

### 1. 问题现象
* **测试报告**：应用在后台灭屏静置时，手机的待机底流（Static Current）偏高，CPU 无法长时间维持在最深级别的 C-State 睡眠中。
* **初步排查**：通过 `dumpsys power` 确认没有任何 Java 层的 WakeLock 被持有。

### 2. 分析步骤与工具链
1. **重置并导出电量统计**：
   运行 `adb shell dumpsys batterystats --reset`。灭屏静置 1 小时后导出数据。
2. **使用 Battery Historian 查看 App 统计**：
   * 选择目标应用包名，在 `App Stats` 中查看 **`Alarms` 触发次数**。
   * 发现该应用在 **1 小时内触发了 40 次闹钟唤醒**，平均每 90 秒就强行唤醒系统一次！
   * 触发的闹钟类型全都是 **`RTC_WAKEUP`**，且类型标记为 **`Exact`（精确闹钟）**。

### 3. 根因剖析（源码诊断）
检查 App 的后台推送及上报模块，发现开发为了绕过 Android 的 Doze（低电耗模式）对后台网络的限制，以实现每 90 秒准时向服务器上报心跳，强行使用了精确闹钟 API：

```java
// 错误示范代码
AlarmManager am = (AlarmManager) getSystemService(Context.ALARM_SERVICE);
// 滥用 setExactAndAllowWhileIdle 绕过 Doze
am.setExactAndAllowWhileIdle(AlarmManager.RTC_WAKEUP, 
    System.currentTimeMillis() + 90000, pendingIntent);
```
* **致命缺陷**：这导致系统无法对这些闹钟执行“对齐批处理”（Batching），CPU 每隔 90 秒就被强行从休眠中唤醒一次。由于射频网络发包又会产生 5~10 秒的“网络尾部高电流时间”，整机几乎一直处于高功耗状态。

### 4. 解决方案
* **后台限制**：Android 14 明确指出禁止非时钟、日程类应用申请 `SCHEDULE_EXACT_ALARM` 权限。
* **重构逻辑**：废弃后台高频心跳上报，替换为 **`JobScheduler`**，设定 Wi-Fi 与充电约束条件。在非充电时，将心跳周期放宽至 15 分钟（非精确对齐闹钟），交由系统统一对齐唤醒：

```java
// 修复后代码
AlarmManager am = (AlarmManager) getSystemService(Context.ALARM_SERVICE);
// 使用非精确闹钟，允许系统合并 Batch，大幅减少唤醒次数
am.set(AlarmManager.RTC_WAKEUP, 
    System.currentTimeMillis() + 15 * 60 * 1000L, pendingIntent);
```

## 案例三：多媒体硬解码与软解码功耗对比（App 播放视频发烫）

### 1. 问题现象
* **测试报告**：用户反馈在使用某款新开发的视频播放 App 观看 1080p 视频时，**手机发热严重，播放 1 小时掉电高达 22%**（同类竞品 App 仅掉电 9%）。

### 2. 分析步骤与工具链
1. **功耗物理电流实测**：
   使用物理程控电源（Power Monitor）测试亮屏播放电流：
   * 播放竞品 App 的 1080p 视频：整机平均电流为 **210mA**。
   * 播放自家 App 的 1080p 视频：整机平均电流高达 **480mA**。
2. **抓取 Perfetto 微观 Trace**：
   使用 Perfetto 录制播放过程 10 秒的 Trace 文件。
3. **分析 Perfetto CPU 负载轨迹**：
   * **竞品 App**：CPU 4 个小核心占用率约为 15%，4 个大核心及超大核心**几乎完全处于 CPUIdle（深睡）状态**。视频渲染顺畅。
   * **自家 App**：CPU 4 个大核心**长期处于 2.2GHz 的最高频运行，使用率高达 75%**，CPU 温度极速攀升。

### 3. 根因剖析
* 自家 App 在集成第三方播放器内核时，为了图省事，默认配置为了 **FFmpeg 软件解码（Soft Decoding）**。
* **软解码**意味着：视频帧的 H.264/H.265 解码算法，全靠 CPU 大核进行高频浮点数学运算硬撑。
* **竞品 App** 则调用了 Android 的 **`MediaCodec` 硬件解码（Hard Decoding）**，直接将视频流投递给 SoC 内部集成的专用多媒体解码芯片（VDEC DSP）。专用多媒体芯片的能效比比通用 CPU 高出数十倍，因此 CPU 可以保持完全休眠。

### 4. 解决方案
重新配置播放器内核，在设备支持时**强制开启 MediaCodec 硬解码**：

```java
// 修复策略示例：配置 ExoPlayer 或自定义解码器开启 MediaCodec
MediaCodecVideoRenderer renderer = new MediaCodecVideoRenderer(
    context, 
    MediaCodecSelector.DEFAULT, // 优先选择硬件解码器
    allowedVideoJoiningTimeMs
);
```
* **优化效果**：开启硬解后，CPU 大核使用率从 75% 瞬间跌落至 $< 5\%$，整机工作电流从 480mA 骤降至 190mA，发热问题彻底解决。
